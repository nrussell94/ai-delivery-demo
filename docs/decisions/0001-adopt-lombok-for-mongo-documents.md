# ADR 0001: Adopt Lombok for `@Document` model classes

- Status: Accepted
- Date: 2026-05-17
- PR: https://github.com/nrussell94/ai-delivery-demo/pull/2

## Context

The `@Document` model classes (`Quote`, `Policy`, `SupportedPostcode`) carry hand-written getters, setters, no-arg constructors and all-args constructors — roughly 70 lines of boilerplate per class for 4-7 fields. The API DTOs are generated by openapi-generator and are not affected by this decision.

This repo demonstrates AI-assisted delivery, where the cost of boilerplate is amplified: every new field requires the model to grow predictably, and the boilerplate is easy to get subtly wrong (e.g. forgetting a setter, mismatching constructor argument order). Deferring the call would mean every future `@Document` class re-litigates the choice.

## Decision

Adopt Project Lombok in `backend/pom.xml` (as `<optional>true</optional>` so it does not leak into downstream consumers), and use it on `@Document` classes with this narrow footprint only:

```
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
```

## Consequences

### Accepted

- ~60 lines of mechanical boilerplate removed per `@Document` class.
- New `@Document` classes follow a uniform shape.
- Lombok's annotation processor is invoked through the maven-compiler-plugin's `annotationProcessorPaths` (the spring-boot-starter-parent default in 3.3.5 already handles this when Lombok is on the classpath).
- Contributors need the Lombok IDE plugin installed (IntelliJ bundles it; VS Code has an extension).

### Explicitly excluded

- **`@Data` is NOT adopted.** `@Data` generates `equals`/`hashCode`/`toString` over **all** fields, which is wrong for Spring Data Mongo entities — equality should be based on `@Id` only, and including the `_id` field in `hashCode` while the document is in a pre-persist state (id == null) breaks set/map membership before save. If a model needs equality semantics, define them by hand or use `@EqualsAndHashCode(of = "id")` explicitly.
- **`@Builder` is NOT adopted by default.** The current call sites construct models via setters or the all-args constructor; adding a third construction pattern is unnecessary.
- **API DTOs continue to use openapi-generator output**, not Lombok. The contract is the source of truth.

### Tooling

- IDE plugin required (one-time install).
- Debugger stepping into Lombok-generated methods is opaque — acceptable trade-off given the methods are trivial.
- Static analysis (e.g. SpotBugs) tends to false-positive on Lombok-generated `equals`; if we adopt static analysis we will configure exclusions.

## Alternatives considered

- **Java 17 `record`** — does not fit because `@Document` mapping requires field mutation by the Spring Data Mongo `MappingMongoConverter` (auditing populates `@CreatedDate` *after* construction).
- **Status quo (hand-written)** — rejected for the reasons in Context.
- **Full `@Data`** — rejected per the Mongo-entity equality footgun.
